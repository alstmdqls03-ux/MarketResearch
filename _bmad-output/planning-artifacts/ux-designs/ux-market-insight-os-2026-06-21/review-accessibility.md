# 접근성 리뷰 — Market Insight OS (WCAG 2.1 AA)

리뷰어 관점: 접근성(WCAG 2.1 AA) · 대상: DESIGN.md / EXPERIENCE.md / color-themes-1.html(Variation 1 = Deep Indigo 채택 테마) · 작성일 2026-06-21

## 총평

스파인의 접근성 의도는 동종 핀테크 평균을 상회한다 — 탭 타깃 ≥44px가 토큰(`spacing.tap-min`)으로 박혀 있고, 색맹 안전을 위해 부호/화살표 글리프 병기가 Do/Don't에 명시되어 있으며, 30초 마찰은 시간제한이 아니라 자가 속도(self-paced)로 설계되어 Timing 위반이 없다. 그러나 **대비(contrast) 층에서 실제로 깨지는 조합이 있다**: 면책 캡션(muted)은 라이트/다크 양쪽에서 4.5:1을 통과하지 못하고, 상승 RED(#E5383B)를 11.5px 소형 헤더 텍스트로 쓰는 순간 4.23:1로 AA 텍스트 기준에 미달한다. 키보드/스크린리더 경로는 "지향" 수준으로만 선언되어 있고 점진 공개의 announce 책임이 컴포넌트 스펙에 묶여 있지 않다 — 의도는 옳으나 계약(contract)으로 못이 박혀 있지 않다.

심각도 집계: **critical 1 · high 3 · medium 4 · low 3**

## 색 대비 (Color Contrast)

측정값은 hex에서 WCAG 상대휘도 공식으로 산출한 근사치다. 표면 기준은 라이트 surface `#FFFFFF`, 다크 surface `#1A1A26`.

**[critical] 면책 고지(disclaimer)가 라이트·다크 양쪽에서 AA 본문 대비에 미달한다.** (DESIGN.md L98-100, `components.disclaimer` = `colors.muted` `#9A9AAE`, caption 11px; color-themes-1.html `.disc` 10px) 라이트 2.76:1, 다크(muted-dk `#6E6E82`) 3.46:1 — 둘 다 4.5:1 미달이고 큰 글자 예외(3:1)도 적용 안 되는 11px 소형 텍스트다. 이건 단순 미관 문제가 아니다: 면책은 **규제 방패이자 제품 원칙**(EXPERIENCE.md L45, FR34)이라 "읽히지 않는 면책"은 법적·제품적 핵심 기능이 접근성으로 깨지는 것이다. 더구나 의도적으로 "본문보다 한 단계 낮은 존재감"(L127)을 노렸기 때문에 디자인 의도와 정면 충돌한다. *Fix:* 면책 텍스트 색을 `muted`에서 분리하라. 라이트는 최소 `text-secondary`(#6B6B80, 5.2:1)급 또는 4.5:1을 통과하는 전용 `disclaimer-fg` 토큰을 신설하고, 다크도 동일 기준으로 산출. 존재감은 색이 아니라 **사이즈·여백·아이콘**으로 낮춰라(대비는 양보 불가). `muted`는 비텍스트 장식(구분선 도트 등)에만 쓰도록 용도를 좁혀라.

**[high] 상승 RED를 소형 텍스트(thesis 헤더·등락률)로 쓰면 AA 미달.** (color-themes-1.html L107-108 `.thesis .h.bull` 11.5px, L92 `.chg` 12.5px; DESIGN.md L167 "상승 블록 헤더 = colors.up") 상승 RED `#E5383B`는 흰 surface 위에서 4.23:1 — 18.66px 미만(굵게 14px 미만) 소형 텍스트의 4.5:1 기준에 미달한다. 하락 BLUE(5.17:1)는 통과하므로 **상승/하락 간 대비 비대칭**이 생긴다. 등락률은 numeric 700이라 "굵은 글자 large(14px+700)" 예외(3:1)에 걸릴 여지가 있으나, **thesis 헤더(11.5px)와 11px caption대 라벨은 예외 없이 실패**한다. *Fix:* 라이트 테마 상승색을 텍스트용으로 한 단계 어둡게 보정한 변형(예: `up-text` ≈ #C42E31대, 4.5:1 확보)을 두거나, 소형 등락/헤더 텍스트는 항상 14px+700 이상으로 강제해 large-text 예외 안에 들도록 타이포 규칙으로 못박아라. "의미색 LOCKED, 절대 변경 금지"(L14, L125)와 충돌하므로, **LOCK은 색상(hue) 정체성에 적용하고 텍스트 가독성용 명도 변형은 허용**한다는 단서를 LOCKED 규정에 추가하라.

**[high] as-of 앰버 배지가 4.51:1로 칼날 통과 — 폰트/안티에일리어싱 변동에 깨질 마진이 없다.** (DESIGN.md L94-96, L170; color-themes-1.html L96-98 `.asof` 10.5px) 라이트 앰버 `#B45309`/`#FEF3C7`는 계산상 4.51:1로 기준을 0.01만 넘긴다. 10.5px 소형 + Pretendard 렌더링 차이 + 서브픽셀에서 실측은 미달로 떨어질 수 있다. as-of 배지는 "데이터가 실시간이 아님"을 알리는 **신뢰 UI의 핵심**(FR32)이라 가독성이 곧 신뢰 기능이다. *Fix:* 앰버 텍스트를 `#9A3E07`대로 한 톤 더 어둡게 내려 5:1대 마진을 확보하라. 배지가 색뿐 아니라 "as of … 실시간 아님" 텍스트를 항상 동반하는 점(L53)은 좋다 — 색 의존이 아니라 텍스트 보강이 있으므로 색-단독 신호 문제는 없음.

**[medium] 도트/보더 등 비텍스트 UI 요소의 3:1 대비가 검증되지 않았다.** (color-themes-1.html L99 `.asof .dot`, L104 `.thesis .blk` 좌측 3px 컬러보더, L101 `.hr` `#E5E5EC`) 좌측 컬러 보더는 상승/하락 구분의 보조 신호인데, 라이트 surface 위 RED 4.23:1·BLUE 5.17:1로 UI 3:1은 통과한다(OK). 그러나 카드 경계인 `border #E5E5EC` / 흰 surface는 약 1.1:1로, 카드 식별을 테두리에만 의존하면 안 된다 — 명세는 "톤 차이 + 1px border"(L152)로 이중화했으니 의도는 맞으나 **border 단독으로 인터랙티브 영역(입력 필드 등)을 구분하는 곳이 없는지** 컴포넌트 단계에서 확인 필요. *Fix:* 입력/포커스 가능한 요소의 경계는 3:1을 충족하는 `input`/`ring` 토큰을 별도 검증하라(shadcn 기본 ring 상속 시 인디고 ring 6.29:1로 안전).

## 색-단독 신호 (Color-as-sole-signal)

**[low] 부호/화살표 병기가 "필수"로 명시되어 있다 — 양호, 단 칩·Score 추세선까지 커버 확인 필요.** (DESIGN.md L181 "방향 색에 부호/화살표 글리프 병기 ▲▼ +/− — 색맹 안전"; EXPERIENCE.md L100) 빨강/파랑 페어 자체는 색각이상에 비교적 안전한 조합이고, 글리프 병기가 Do/Don't에 못박혀 있으며 HTML 목업도 `▲ +3.92` 형태로 준수한다. 다만 **Score 거울의 추세 라인/바**(DESIGN.md L172 "추세는 단순 라인/바")가 상승/하락 색만으로 방향을 전달하면 색-단독이 된다. *Fix:* 추세 시각화에도 직접 라벨(+/− 값) 또는 패턴(실선/점선)을 병기하도록 Score Mirror 스펙에 한 줄 추가하라. 현 스파인은 "색만으로 전달 금지"를 카드 등락에는 명시했으나 추세 그래프에는 미적용.

## 탭 타깃 (Touch Targets)

**[medium] ≥44px가 칩·버튼에는 적용됐으나, "기타(직접)" 타이핑 전환·삭제·미세 인터랙션의 타깃 크기가 누락.** (DESIGN.md L75 tap-min 44px, L86 chip min-height, L93 button 46px; EXPERIENCE.md L98) 칩(44px)·Primary 버튼(46px)은 명시적으로 통과. 그러나 관심종목 "삭제"(EXPERIENCE.md L26 FR3), as-of 배지/면책 옆의 ⓘ 아이콘 탭(상세 펼침이 있다면), 칩 내부 "기타" 텍스트 입력 모드의 close/back 컨트롤은 크기 규정이 없다. 모바일에서 삭제·실행취소 같은 보조 액션이 44px 미만이면 WCAG 2.5.5(Target Size, AAA지만 모바일 실사용 필수)에서 깨진다. *Fix:* "모든 인터랙티브 요소(보조·파괴 액션 포함) ≥44px"로 일반화하고, 아이콘 전용 버튼은 시각 크기와 별개로 히트영역 44px를 명시하라.

## 키보드 / 스크린리더 (Keyboard / Screen Reader)

**[high] 비타이핑(탭/칩) 경로의 포커스 순서·레이블·점진 공개 announce가 "지향" 선언에 그쳐 컴포넌트 계약으로 박혀 있지 않다.** (EXPERIENCE.md L97-102 Accessibility Floor; L86 progressive disclosure) 핵심 루프가 탭/칩 기반이라 키보드/SR 운용성이 곧 핵심 기능 운용성이다. 현 스파인은 "WCAG 2.1 AA 지향", "Tab 순서는 읽기 순서", "Esc는 모달 닫기"까지만 적었고 — **점진 공개에서 "감정 탭 → '계획인가 감정인가' 질문 → 나머지 필드가 펼쳐지는"** 동적 콘텐츠가 SR에 어떻게 announce되는지(예: `aria-live`, 포커스 이동) 규정이 없다. 칩의 selected 상태(현재 색으로만 구분, DESIGN.md L168)가 `aria-pressed`/`role` 없이 색에만 의존하면 SR 사용자에게 선택 상태가 전달되지 않는다. *Fix:* (1) 칩 = `role="button"` + `aria-pressed` 또는 토글 그룹 `aria-checked`로 선택 상태를 프로그램적으로 노출. (2) 점진 공개로 펼쳐지는 영역은 `aria-live="polite"` 또는 펼침 직후 새 필드 첫 요소로 포커스 이동, "계획인가 감정인가" 마이크로 개입은 announce되도록 명시. (3) 이 세 항목을 EXPERIENCE.md Component Patterns의 칩/점진공개 행위 규칙에 계약으로 추가.

**[medium] "기타(직접)" 타이핑 탈출구는 비타이핑 경로의 완전 운용성을 깨지 않는다 — 검증 OK, 단 입력 모달의 포커스 트랩/복귀 미규정.** (EXPERIENCE.md L64, L87) 모든 필수 입력이 탭/선택으로 완료 가능하고 타이핑은 항상 선택이라(NFR-U1) 비타이핑 경로 운용성은 설계상 보장된다 — 좋다. 다만 "기타" 선택 시 타이핑 입력으로 전환되는 순간 포커스가 입력 필드로 가고, 취소 시 원래 칩으로 복귀하는 포커스 관리가 없다. *Fix:* 칩→입력 전환 시 포커스 이동과 Esc 취소 후 포커스 복귀를 칩 행위 규칙에 명시.

**[low] 등락 SR 낭독 규칙은 있으나("상승/하락 + 수치"), 숫자·티커의 SR 발음/언어 태깅 미규정.** (EXPERIENCE.md L101) "상승/하락 + 수치"로 읽도록 한 점은 색-단독을 SR에서도 보강하는 좋은 결정이다. 다만 미국 티커(NVDA)와 한국어가 한 줄에 섞이고(DESIGN.md L134) tabular-nums 수치가 많아, `$172.41`·`-8%` 같은 값이 SR에서 자연스럽게 읽히도록 `lang` 속성/`aria-label`로 보강할 여지가 있다. *Fix:* 혼합 언어 라인에 부분 `lang` 태깅, 등락 값은 "상승 2.3퍼센트"처럼 읽히는 `aria-label` 패턴을 권장으로 추가.

## 모션 / 타이밍 (Motion / Timing)

**[low] 30초 "과속방지턱"은 자가 속도이며 강제 시간제한이 없다 — WCAG 2.2.1 위반 없음. 확인 완료.** (EXPERIENCE.md L88 "30초는 0으로 최적화하지 않는다 … <100ms지만 흐름을 0초로 줄이지 않는다") 명시적으로 "강제 모달 게이트·행동 차단형 경고 금지"(L91)이고, 30초는 사용자가 멈춰 생각하게 하는 의도적 마찰일 뿐 카운트다운·자동 진행·세션 만료가 아니다. 따라서 2.2.1 Timing Adjustable 대상이 아니다. 깜빡임 애니메이션도 금지(L91)되어 2.3.1(Three Flashes) 위험도 차단됐다. *Fix:* 불필요. 다만 향후 "넛지"가 자동으로 사라지는 토스트라면 Toast의 표시 시간이 충분하거나 dismiss 가능해야 함을 Toast 행위에 한 줄 명시하면 완결.

## 피드백 (Feedback)

**[medium] 100ms 피드백·가시적 오류는 좋으나, 오류·재시도 상태가 SR에 announce되는지 미규정.** (DESIGN.md L173 async-state; EXPERIENCE.md L69, L77 "조용히 넘어가지 않음 + 재시도") 실패를 침묵하지 않고 `destructive` 오류 + 재시도 버튼을 가시화하는 것은 SR 사용자에게도 유리한 올바른 방향이다. 다만 "가시적"이 시각만 의미하면 SR 사용자는 오류 발생을 모를 수 있다. 비동기 생성(수 초)이 실패로 끝나는 순간이 특히 위험하다. *Fix:* 오류 상태는 `role="alert"` 또는 `aria-live="assertive"`로 announce하고, 진행 표시(Skeleton/progress)는 `aria-busy`로 상태를 노출하도록 async-state 행위 규칙에 명시. 100ms 칩 반영도 SR에서 선택 상태 변화가 전달되는지(앞의 `aria-pressed`와 연계) 확인.

## 우선 조치 요약

1. **[critical]** 면책 muted 색 분리 — 라이트/다크 모두 4.5:1 확보(규제·제품 핵심 기능).
2. **[high]** 상승 RED 텍스트용 명도 변형 도입(소형 텍스트 4.5:1) — LOCKED 규정에 "hue는 고정, 텍스트 명도 변형 허용" 단서 추가.
3. **[high]** 칩 selected 상태 `aria-pressed` + 점진 공개 `aria-live`/포커스 이동을 컴포넌트 계약으로 박기.
4. **[high]** as-of 앰버를 5:1대로 한 톤 강화(현 4.51:1 칼날).
